## WHAT "ONE BASE, MANY PLATES" MUST DO — UNDERSTAND THIS FIRST

Before running any test, read the implementation and confirm it does ALL of the following:

**The core rule:** One base dish is generated for the whole household. The algorithm then routes each family member to a personalised version of that dish by injecting step-level modifications. The base dish is cooked once. The modifications happen as cooking branches, not as separate meals.

**The algorithm must handle these operations in order:**

```
Step 1 — BASE SELECTION
Given a meal slot (breakfast / lunch / dinner) and the household dietary baseline,
select one base dish that satisfies the most members without violating any Level-1
or Level-2 constraint (allergy or religious rule) for any member.

Step 2 — PULL-BEFORE OPERATIONS
Identify members who must receive their portion BEFORE a later ingredient is added.
Example: Dadi follows Jain rules — her portion is pulled before onion/garlic goes in.
Example: Sunita is allergic to dairy — her portion is pulled before ghee/butter is added.
These are time-critical kitchen events. They must appear as numbered steps with ⏱️ markers.

Step 3 — QUANTITY MODIFIERS
For members with clinical conditions, calculate adjusted quantities for specific ingredients.
Example: Rajesh (hypertensive + diabetic) — salt reduced to 50%, ghee reduced to 25%,
rice portion reduced to 150g instead of 250g.
These are injected as inline annotations inside the cooking step, not as a separate section.

Step 4 — ADDITIVE OPERATIONS
Some members get additional components added to their plate after the base is served.
Example: Arjun (muscle building) — extra dal protein scoop added on top.
Example: Anaemic member — sprinkle of sesame seeds or amla chutney on the side.
These appear as PLATE ASSEMBLY instructions at the end, per member.

Step 5 — FULL PLATE DESCRIPTION
For each member, the algorithm must produce a final plate summary:
- What they receive
- What was modified
- What was added
- What was withheld
- Estimated macros for their specific plate (calories, protein, carbs, fat, sodium, iron if relevant)

Step 6 — CONFLICT ESCALATION
If no single base dish can satisfy Step 1 (e.g. one member requires non-veg,
another is strictly Jain with no sharing kitchen), the algorithm must NOT silently fail.
It must return a ConflictEscalation object explaining why and proposing the minimum
number of parallel dishes needed (never more than 2 for cost efficiency).
```

The return type must be:

```typescript
interface OneManyPlatesResult {
  baseDish: string;
  baseInstructions: CookingStep[];   // numbered steps for the base dish
  pullBeforeEvents: PullEvent[];     // timed kitchen branch events
  memberPlates: MemberPlate[];       // one per family member
  conflictEscalation?: ConflictEscalation;  // only if Step 1 fails
  harmonyDeductions: HarmonyDeduction[];    // for the Harmony Score
}

interface CookingStep {
  stepNumber: number;
  instruction: string;
  pullBefore?: string[];    // member names who must get their portion before this step
  memberAnnotations?: {     // inline per-member modifications at this step
    memberName: string;
    modification: string;
  }[];
  timerSeconds?: number;
}

interface MemberPlate {
  memberName: string;
  receives: string;         // plain English description of their plate
  withheld: string[];       // ingredients not in their plate
  added: string[];          // extras added to their plate
  modified: string[];       // quantities changed
  estimatedMacros: {
    calories: number;
    proteinG: number;
    carbsG: number;
    fatG: number;
    sodiumMg: number;
    ironMg?: number;
  };
  warningFlags: string[];   // any clinical alerts for this plate
}

interface PullEvent {
  beforeStep: number;
  memberNames: string[];
  reason: string;
  urgency: "CRITICAL" | "RECOMMENDED";
}

interface ConflictEscalation {
  reason: string;
  affectedMembers: string[];
  proposedSolution: string;
  parallelDishesNeeded: number;
}

interface HarmonyDeduction {
  reason: string;
  points: number;
}
```

If the implementation exists but does not match this structure, refactor it to match before running tests.
