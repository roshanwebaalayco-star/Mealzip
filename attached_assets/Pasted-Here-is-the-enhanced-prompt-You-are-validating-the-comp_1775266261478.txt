## THE PIPELINE YOU ARE VALIDATING

This is the exact architecture. Every step must exist and work in this order:

```
ICMR Rules (hardcoded TypeScript constants)
    ↓
Family Profile + Weekly Context (read from Supabase)
    ↓
Conflict Engine (pure TypeScript, < 1 second, no AI)
    ↓ output: one MedicalLaw object per family member
    
12,000 recipes in Supabase recipes table
    ↓ SQL filter: region + diet type + allergens + cook time
    ↓ output: 80–120 recipe name strings only (no steps, no ingredients)
    
Gemini Call 1 — The Scheduler (3–12 seconds)
    ↓ input: the 80–120 recipe names + MedicalLaw objects + budget constraints
    ↓ output: strict JSON — 21-meal calendar with memberModifiers per meal
    ↓ Gemini NEVER outputs cooking steps here. Names and modifiers only.
    
Store calendar JSON in Supabase meal_plans table
    ↓
Return 7-day schedule to frontend (total pipeline < 90 seconds)

--- SEPARATELY, LATER ---

User taps a specific meal
    ↓
Gemini Call 2 — The Recipe Generator (3–5 seconds)
    ↓ input: meal name + that member's modifiers from the stored calendar
    ↓ output: full recipe with steps, quantities, and inline member warnings
    ↓ Gemini only formats. Clinical decisions were already made by Conflict Engine.
```

## STEP 1 — AUDIT THE IMPLEMENTATION

Before writing any test, audit these files and confirm each exists and is correctly implemented. For each file, state what you found and what (if anything) you fixed:

**A. ICMR Rules file**
Find the file where ICMR dietary rules are hardcoded as TypeScript constants. It must contain rules for at minimum: Type-2 Diabetes (low GI, max sugar per meal), Hypertension (sodium cap per day), Anaemia (iron-rich food list, calcium conflict window), Iron Supplements (no dairy within 1 hour), Metformin (no high-sugar meal within 2 hours of dose time), Levothyroxine (meal must be 30–60 min after medication), Warfarin (no sudden spike in Vitamin K foods). If this file does not exist, create it at `server/lib/icmrRules.ts`. If it exists but is incomplete, add the missing rules.

**B. Conflict Engine**
Find the function that takes a family member's profile (conditions array, medications array, timings array) and returns a `MedicalLaw` object. It must be pure TypeScript with zero AI calls. It must be deterministic — same input always produces same output. If it does not exist, create it at `server/lib/conflictEngine.ts`. The output shape must be:
```typescript
interface MedicalLaw {
  memberId: number;
  memberName: string;
  hardBlocks: string[];        // ingredients completely forbidden
  softBlocks: string[];        // ingredients to minimise
  mealTimingRules: string[];   // e.g. "breakfast must follow Levothyroxine by 45 min"
  portionModifiers: string[];  // e.g. "sodium cap 1200mg/day"
  priorityLevel: 1 | 2 | 3 | 4 | 5 | 6;
}
```

**C. Recipe SQL Filter**
Find the database query that filters the recipes table. It must filter simultaneously by: dietary type (veg / non-veg / eggetarian / Jain), allergens (exclude recipes containing any allergen in the family's combined allergen list), region (match family's state), cook time (match weekday or weekend availability). The query must return only recipe name and id — not ingredients, not steps. This keeps the Gemini prompt small. If the query fetches full recipe objects, fix it to select only `id` and `name`.

**D. Gemini Scheduler (Call 1)**
Find the function that calls Gemini with the recipe shortlist and MedicalLaw objects. Confirm: the system prompt forbids Gemini from inventing recipe names not in the provided list. The output must be parsed as JSON before storing. The JSON shape must be validated — if Gemini returns malformed JSON or a recipe name not in the shortlist, the call must retry once, then throw a recoverable error. If this validation does not exist, add it.

**E. Meal Plan Storage**
Confirm the 21-meal calendar JSON is written to the `meal_plans` table in Supabase with at minimum: `familyId`, `weekStartDate`, `calendarJson` (the full 21-meal object), `createdAt`.

**F. On-Demand Recipe Generator (Call 2)**
Find the function triggered when a user taps a meal. Confirm: it reads the stored `memberModifiers` from the calendar JSON in Supabase — it does NOT rerun the Conflict Engine. It passes the meal name and modifiers to Gemini. Gemini outputs a recipe with inline warnings injected into the steps (e.g. "Step 3: Add salt. ⚠️ Papa: use half quantity"). If the modifiers are not being injected into the Gemini prompt, fix it.
