export interface ComputedMemberPlate {

  harmony_deduction_reason: string | null;

  slot: string;

**Internal logic — implement all six steps in order:**

**Step A — HARD BLOCK DETECTION per member per base ingredient**

The algorithm does not know the actual ingredients of a recipe at this stage (Gemini has not picked the recipes yet). Instead it computes a PROFILE of what each member cannot receive, and returns these as modifier strings. Gemini then applies them when it selects a recipe.

For each `EffectiveMemberProfile` in `packet.effectiveProfiles`:

1. Collect all hard-blocked ingredient categories from:
   - `ALLERGY_INGREDIENT_MAP[allergy]` for each of their allergies
   - `RELIGIOUS_FORBIDDEN_MAP[profile.religiousCulturalRules.type]`
   - Any `forbidden_ingredients` from their `activeMedications` via `packet.medicationGuardrailBundles`

2. Collect soft-limited ingredients from:
   - Condition-specific limits (e.g. diabetes → limit white rice, limit potato)
   - Ingredient dislikes

3. Generate modification strings from hard blocks. Examples:
   - Allergy (dairy): `"ALLERGY CRITICAL: Pull Kavita's portion before any dairy (ghee/paneer/curd) is added"`
   - Jain: `"JAIN RULE: Pull Dadi's portion before onion and garlic tempering step"`
   - Religious no_beef: `"RELIGIOUS: Ensure no beef in Ammachi's plate — serve from the vegetarian preparation"`

4. Generate modification strings from soft limits. Examples:
   - Diabetes + white rice: `"DIABETES: Reduce Rajesh's rice to 150g (60% of standard portion). Substitute with jowar roti if available"`
   - Hypertension: `"HYPERTENSION: Rajesh's portion — use half the salt called for in the base recipe. Add lemon juice to compensate"`
   - Obesity: `"OBESITY: Sunita's portion — no ghee, steam/grill separately if fried in base"`

**Step B — PULL_BEFORE EVENT COMPUTATION**

A pull-before event is needed when a member's portion must physically leave the pot before a later ingredient makes the food unsuitable for them.

Compute pull_before_step using this priority table (step numbers are approximate — Gemini will map them to actual recipe steps):
- Jain/sattvic member in household that cooks with onion/garlic: pull before step 3 (tempering), urgency CRITICAL
- Dairy-allergic member when base dish uses ghee, paneer, or curd: pull before step where dairy is first added, urgency CRITICAL
- Non-veg base dish with a vegetarian member: this is a ConflictEscalation, not a pull-before
- Member with hypertension where base dish adds >500mg sodium: pull before final salting step, urgency RECOMMENDED

If a pull-before event is needed, set `pull_before_step`, `pull_before_reason`, and `pull_before_urgency` on the member's plate. Add this to their modifications array: `"⏱️ PULL BEFORE STEP [N]: [reason]"`.

**Step C — ADDITIVE COMPUTATION**

Additives are ingredients added only to specific members' plates after the base dish is served.

Add additives for:
- Member with anaemia → `"ADD: Squeeze lemon on Sunita's plate (Vitamin C enhances iron absorption from the dal/sabzi)"` + `"ADD: Sprinkle til/sesame on Sunita's plate if dish permits"`
- Member with goal build_muscle → `"ADD: Extra dal scoop (protein boost) on Arjun's plate"` + if non-veg is allowed: `"ADD: Grilled chicken/egg on the side for Arjun"`
- Member with PCOS → `"ADD: Half tsp ground flaxseed (alsi) on Divya's plate for omega-3"`
- Member with high_cholesterol → `"ADD: Small portion of walnuts on side if dish permits"`
- Senior 60+ with low calorie target → `"ADD: Small portion of fortified cereal or makhana on side to maintain minimum 1400 kcal"`
- Child under 5 → `"TEXTURE: Mash/blend Rohan's portion separately. Reduce spice to zero"`

**Step D — FASTING REPLACEMENT**

If a member is fasting on this day AND a festival is active:
- Check `packet.weeklyContext.activeFestival` (e.g. "Navratri")
- Replace their meal entirely: set `modifications` to `["FASTING DAY: Replace [member]'s [slot] with [festival-appropriate dish]. See fasting_replacement field."]`
- High GI fasting foods warning: if festival allows sabudana and member has diabetes, add `"⚠️ DIABETES + FASTING: Sabudana is high GI (70). Limit to 80g cooked. Add peanuts (if allowed) to slow glucose spike. Consider kuttu roti instead."`

**Step E — CONFLICT ESCALATION**

Set `base_dish_is_valid = false` when:
- One or more members are non-vegetarian and one or more are strictly vegetarian AND the non-veg ingredient is in the base dish (not just an additive). Proposed solution: one base dish, with separate protein preparation — vegetarian base + non-veg protein track for specific members
- One member is Jain and the base dish is fundamentally incompatible (e.g. cannot make the dish without root vegetables, onion, garlic) — Proposed solution: parallel simplified dish for Jain member using approved ingredients
- Kidney patient and the base dish is the primary high-potassium/phosphorus food for that meal (e.g. Rajma for a kidney patient as the ONLY option)

When escalating, set:
- `parallel_dishes_needed: 2` (always 2 — never 3, for cost efficiency)
- `escalation_reason`: specific clinical or dietary reason
- `escalation_solution`: exactly what the two parallel preparations should be
- Add to harmony deductions: -5 points for each escalation

**Step F — MACRO TARGETS PER MEMBER PER SLOT**

For each member and slot, compute:
- `calories`: `profile.dailyCalorieTarget × mealSlotWeight[slot]`
- `protein_g`: based on goal — build_muscle: 2g/kg weight, others: 0.8g/kg weight, per slot
- `carbs_g`: (calories × 0.55) / 4 for standard; (calories × 0.40) / 4 for diabetes
- `fat_g`: (calories × 0.25) / 9
- `sodium_mg_max`: from condition (hypertension: 1500mg/day → 500mg/slot, kidney: 1200mg/day → 400mg/slot, default: 2000mg/day)

Add a clinical flag if any macro target is clinically significant:
- `"SODIUM CAP: Rajesh's plate must not exceed 500mg sodium in this meal"`
- `"HIGH PROTEIN: Arjun needs 40g+ protein at this meal to meet muscle-building target"`

```typescript
// Build modifier injection string from the pre-computed map
function buildModifierInjectionSection(
  modifierMap: MemberModifierMap,
  weekStartDate: string
): string {
  const lines: string[] = [
    "PRE-COMPUTED PLATE MODIFICATIONS — YOU MUST USE THESE EXACTLY.",
    "These modifications were computed deterministically by the clinical engine.",
    "Do NOT change, omit, or add clinical modifications of your own.",
    "Your job is to assign recipes and format these into the member_plates array.",
    "",
  ];

  for (const [key, output] of modifierMap.entries()) {
    lines.push(`[${key.toUpperCase()}]`);
    if (!output.base_dish_is_valid) {
      lines.push(
        `  ⚠️ CONFLICT ESCALATION: ${output.escalation_reason}`,
        `  SOLUTION: ${output.escalation_solution}`,
        `  Assign two parallel preparations. Use parallel_dish: true on the conflicted member's plate.`
      );
    }
    for (const plate of output.member_plates) {
      lines.push(`  ${plate.member_name} (${plate.member_id}):`);
      plate.modifications.forEach((m) => lines.push(`    - ${m}`));
      if (plate.additives.length > 0) {
        lines.push(`    ADDITIVES: ${plate.additives.join(" | ")}`);
      }
      if (plate.pull_before_step !== null) {
        lines.push(`    ⏱️ PULL BEFORE STEP ${plate.pull_before_step}: ${plate.pull_before_reason}`);
      }
      if (plate.clinical_flags.length > 0) {
        lines.push(`    ⚠️ FLAGS: ${plate.clinical_flags.join(" | ")}`);
      }
    }
    lines.push("");
  }

  return lines.join("\n");
}
```
